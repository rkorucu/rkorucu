Agentic Claim Decision Engine

An AI-powered insurance claim decision platform using a hybrid deterministic + generative architecture for enterprise-grade decision automation.

View Repository

Problem

Insurance claim adjudication typically relies on either fully manual review (slow, inconsistent) or rigid rule engines (brittle, unable to explain decisions). The industry needs a system that combines deterministic fraud/coverage analysis with optional LLM-powered explanation and self-checking — while remaining fully operational without any AI dependency.

Architecture

Three-service architecture orchestrated via Docker Compose:

  • Spring Boot API Gateway (:8080) — Receives claim requests from the frontend, validates input structure, and proxies analysis to the agent service. Handles CORS, health checks, and response mapping.
  • FastAPI Agent Service (:8000) — Executes the agentic workflow: deterministic tool pipeline → optional LLM explanation → optional reflection loop. Stateless by design — each request carries its own full context.
  • React Dashboard (:5173) — Single-page interface with claim submission form, decision card, flags table, and reflection panel. Renders structured JSON responses into human-readable decision reports.

How It Works — Execution Lifecycle

  1. Claim Ingestion — User submits a claim via the React dashboard or API. The Spring Boot gateway validates the payload schema and forwards to the agent service.
  2. Deterministic Tool Pipeline — Three structured tools execute sequentially: coverage_check (validates policy constraints), fraud_signal_scoring (scores fraud indicators), and risk_score_aggregation (combines signals into a composite risk score with decision thresholds).
  3. Draft Decision — Based on aggregated risk score and flag analysis, a deterministic decision is produced: APPROVE, REVIEW, or REJECT — with full traceability of which flags triggered which thresholds.
  4. LLM Explanation (optional) — If an OpenAI API key is configured, the decision context is sent to GPT for a human-readable explanation paragraph. The LLM receives only structured decision data — never raw claim text.
  5. Reflection Loop (optional) — A second LLM call self-checks consistency: Does the decision match the score? Does the risk level align with thresholds? Is the explanation coherent and non-contradictory? Adjustments or reflection notes are appended.
  6. Response — The complete ClaimDecision (decision, riskScore, riskLevel, flags, explanation, reflectionNotes) is returned as validated JSON.

AI / Tool Integration

  • Agent Roles — The agent service acts as a single orchestrator with a fixed tool pipeline. No autonomous tool selection — tool execution order is deterministic and pre-defined.
  • Reflection Loop — Self-checking step validates internal consistency between score, decision, risk level, and explanation. Contradictions trigger field adjustments and reflection notes.
  • Deterministic Tool Grounding — All fraud scoring, coverage checking, and risk aggregation use structured, testable Python functions — not LLM-generated logic. The LLM is constrained to explanation and reflection only.
  • JSON Validation — Input claims are validated against Pydantic schemas. Output decisions are type-validated before response serialization.
  • State Handling — Stateless request processing. Each claim carries its own complete context. No session state, no cross-request dependencies.
  • Safety Constraints — Graceful fallback mode: when no OpenAI key is configured, steps 2 and 3 are skipped entirely. The system returns a fully valid deterministic-only decision with a clear fallback indicator.

Reliability Measures

  • Schema Validation — Pydantic models enforce strict input/output contracts. Invalid payloads are rejected before entering the pipeline.
  • Guardrails — LLM outputs are post-processed and validated against expected structure. The system never surfaces raw LLM text — only structured, validated fields.
  • Controlled Iteration — The reflection loop runs exactly once. No recursive self-correction chains that could amplify errors or hallucinate fixes.
  • Observability — Health check endpoints on both services. Structured logging across the tool pipeline. Decision audit trail embedded in every response.

SkillBridge — AI-Powered Expert Matching Platform

An AI-driven expert marketplace using semantic embeddings, LLM-powered matching, and automated session intelligence to connect clients with specialized professionals.

View Repository

Problem

Traditional expert marketplaces rely on keyword-based search and manual filtering, which fails to capture the nuance of expertise matching. A client describing "I need help optimizing database queries for a multi-tenant SaaS application" won't match an expert tagged with "SQL" and "PostgreSQL" — even though the expertise is directly relevant. The system needs semantic understanding, not keyword lookup.

Architecture

Full-stack application with AI-powered matching layer:

  • Kotlin + Spring Boot Backend — RESTful API layer handling user management, session orchestration, and expert profile management. Clean domain separation between matching logic, user state, and session lifecycle.
  • AI Matching Engine — Semantic embedding pipeline that converts both client requests and expert profiles into vector representations, enabling similarity-based matching that understands context beyond keywords.
  • React + TypeScript Frontend — Type-safe client with component-driven architecture. Expert discovery, session booking, and AI-generated session summaries rendered through a responsive, accessible interface.
  • PostgreSQL Data Layer — Relational data model supporting multi-dimensional expert profiles, session history, and matching metadata with query-optimized indexing.

How It Works — Execution Lifecycle

  1. Profile Ingestion — Expert profiles are ingested with structured metadata (skills, domains, experience level) and converted into semantic embeddings for the matching index.
  2. Client Request Analysis — Client queries are processed through the LLM to extract intent, required expertise dimensions, and context — producing a structured matching request.
  3. Semantic Matching — The matching engine computes similarity scores between the client's request embedding and the expert profile embeddings, ranking candidates by relevance.
  4. Session Orchestration — Matched expert-client pairs enter a session lifecycle: scheduling, active session, and post-session processing.
  5. Automated Session Intelligence — Post-session, the system generates structured summaries capturing key discussion points, action items, and follow-up recommendations.

AI / Tool Integration

  • Semantic Embeddings — Expert profiles and client requests are converted into dense vector representations, enabling similarity matching that captures contextual expertise alignment.
  • LLM-Powered Matching — Large language models extract structured intent from free-text client queries, bridging the gap between natural language requests and structured expert metadata.
  • Automated Summaries — Post-session AI generates structured summaries with key takeaways, action items, and follow-up recommendations — reducing manual note-taking overhead.
  • Validation — Matching results are scored and ranked, with confidence thresholds ensuring minimum quality before surfacing matches to clients.

Reliability Measures

  • Type-Safe Contracts — Kotlin's type system and TypeScript's static analysis enforce contract integrity across the full stack.
  • Data Integrity — PostgreSQL constraints, migrations, and transactional session management ensure consistent state across matching, booking, and session lifecycle.
  • Graceful Degradation — Matching fallback to keyword-based search when embedding services are unavailable. Session summaries degrade to manual templates when AI services are down.

System Design Philosophy

State-Driven Design

Every system I build has an explicit state model. Components transition between well-defined states with validated inputs and outputs at each boundary. If the system can't tell you exactly what state it's in and why — the design is incomplete.

Deterministic + Generative Hybrid

LLMs are powerful, but unreliable when unconstrained. I design systems where deterministic tools handle the critical path — scoring, validation, decision logic — and generative AI is sandboxed to explanation, summarization, and reflection. The system works without AI. AI makes it better.

Failure Mode Thinking

Every integration point, every external service call, every state transition has a defined failure mode. Graceful degradation is not an afterthought — it's a design constraint. Systems must produce correct, if reduced, output when dependencies fail.

Hallucination Risk Control

In AI-integrated systems, hallucination is a reliability failure, not a feature limitation. I mitigate it through structured output schemas, deterministic tool grounding, post-generation validation, reflection-based consistency checks, and strict separation between generative and authoritative data paths.

Observability as a First-Class Concern

If I can't observe it, I can't trust it. Every system I build includes structured logging, health checks, execution tracing, and decision audit trails. In AI systems especially, observability is what separates a prototype from a production platform.
